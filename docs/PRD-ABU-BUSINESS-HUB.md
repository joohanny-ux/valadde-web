## 1. 프로젝트 개요

### 1.1 목적
관리자가 업로드한 상품을 한곳에 게시하고, **크리에이터·바이어·브랜드**가 각자 목적에 맞게 이용하는 **랜딩/비즈니스 허브**를 구축한다.

### 1.2 한 줄 요약
> “상품 DB를 중심으로, 크리에이터(방송 판매)·바이어(B2B)·브랜드(검수·제안)가 쓰는 랜딩 및 비즈니스 허브”

### 1.3 배포 계획
| 단계 | 대상 | 비고 |
|------|------|------|
| 1차 | 웹 (www.valadde.com) | 반응형 웹 우선 |
| 2차 | 앱 (iOS/Android) | 웹 안정화 후 진행 |
| 다국어 | 한글 → 영어 → 중국어 | 단계별 롤아웃 |

## 2. 사용자 정의 및 시나리오

### 2.1 사용자 유형

| 유형 | 설명 | 핵심 행동 |
|------|------|------------|
| **크리에이터** | 라이브/쇼츠로 상품 판매 | 상품 검색·선택 → **판매 의사·재고·가격 협상** → 주문·구매 |
| **바이어** | B2B 수출·대량 구매 | 상품 검색·선택 → **PO(Purchase Order)** → 구매 |
| **브랜드** | 상품 검수·신상 등록 요청 | 기존 상품 가격/페이지 검수 요청, 신상 게시 요청 → (차후) 크리에이터 검색·판매 제안·매칭 |
| **관리자** | 플랫폼 운영 | 상품/유저/주문 DB, 인보이스·결제·배송, 게시판·CRM·통계·약관 관리 |

### 2.2 크리에이터 플로우 (요약)
1. 상품 검색·필터
2. 관심 상품 선택 → **판매 의사 신청**
3. 재고 현황 확인, **판매가·공급가 협상**
4. 협상 완료 후 주문·결제·배송

### 2.3 바이어 플로우 (요약)
1. 상품 검색·필터
2. 상품 선택 후 **PO 작성·제출**
3. 주문·인보이스·결제·배송

### 2.4 브랜드 플로우 (요약)
1. 기존 상품 가격/상세페이지 **검수 요청** 전달
2. **신상품** 게시 요청 (관리자 검토 후 반영)
3. (차후) 크리에이터 검색·판매 제안 → 본사 매칭

## 3. 개발 단계 및 권장 순서

### 3.1 제안하는 개발 순서 (개념)

```
[기획·정의] → [백엔드·DB] → [관리자 대시보드] → [퍼블릭 프론트엔드] → [다국어·앱]
```

**이유 (초보자·1인 개발 관점)**  
- **DB·API 먼저**: 화면을 만들기 전에 “어떤 데이터를 어떻게 줄지”를 정해야 화면 설계가 명확해짐.  
- **관리자 대시보드 먼저**: 상품·유저·주문을 직접 넣고 관리할 수 있어야, 프론트에서 보여줄 데이터가 생김.  
- **랜딩/메인·서브는 그 다음**: 데이터 구조와 관리자 플로우가 잡힌 뒤에 메뉴·디자인·사용자별 노출을 설계하는 것이 효율적.

### 3.2 단계별 상세

| 단계 | 내용 | 산출물 |
|------|------|--------|
| **0. 기획** | PRD 확정, 화면 목록, API·DB 스키마 초안 | PRD, 와이어프레임/목록 |
| **1. 백엔드** | Product DB → Accounts DB → Order DB 순으로 설계·구현 | Cloud DB, API |
| **2. 대시보드** | Admin 로그인, 상품/유저/주문/게시판/CRM/통계/약관 | Admin 웹 앱 |
| **3. 프론트엔드** | 메인·서브 페이지, 메뉴·디자인·컨텐츠·사용자별 노출 | 랜딩·서비스 페이지 |
| **4. 다국어·앱** | i18n 적용, (선택) PWA 또는 네이티브 앱 | 다국어 웹, 앱 |

## 4. 백엔드 설계 (Google Cloud 기준)

### 4.1 Google Cloud에서 DB·백엔드 가져오는 방법 (초보자용 설명)

- **Firestore**  
  - 문서형 NoSQL. 스키마를 유연하게 가져가기 좋고, 실시간 동기화·클라이언트 SDK 제공.  
  - **장점**: 설정이 상대적으로 쉽고, 소규모~중규모에 적합.  
  - **단점**: 복잡한 집계·통계는 쿼리 제한이 있어 별도 처리 필요.

- **Cloud SQL (MySQL/PostgreSQL)**  
  - 전통적인 관계형 DB. 주문·인보이스·결제처럼 관계가 많은 데이터에 적합.  
  - **장점**: SQL에 익숙하면 설계·통계 쿼리 작성이 직관적.  
  - **단점**: 인스턴스 관리·백업 등 운영 부담이 Firestore보다 큼.

- **권장 (1인·초기 서비스)**  
  - **상품·카테고리·브랜드**: Firestore로 시작 (빠른 개발, 확장 용이).  
  - **주문·인보이스·결제·배송**: 구조가 정해지면 Cloud SQL 도입 검토 가능.  
  - 또는 **Firestore만** 사용하고, 주문/결제는 “문서 단위 설계”(주문 1건 = 1문서, 서브컬렉션으로 상품·배송)로 가져가는 것도 가능.

### 4.2 DB 구성 (개념)

#### Product DB (우선)
- 상품 마스터: 이름, 카테고리, 브랜드, 이미지, 가격, 재고, 상세페이지 URL/내용 등.
- 카테고리·브랜드: 코드/이름 등 (필요 시 별도 컬렉션 또는 문서).
- **가져오는 방법**:  
  - 수백 개는 스프레드시트 → CSV/JSON으로 정리 후, **스크립트(예: Node/Python)**로 Firestore에 일괄 업로드.  
  - 이후에는 관리자 대시보드에서 등록·수정.

#### Accounts DB (차후)
- 크리에이터/바이어/브랜드 계정: 이메일, 비밀번호(해시), 유형, 프로필, 연락처 등.
- Firebase Authentication + Firestore(프로필·권한) 조합을 많이 사용.

#### Order DB (차후)
- 주문, 인보이스, 결제, 배송 상태.
- 크리에이터용 “판매 의사·협상” 이력도 여기 또는 별도 컬렉션으로 관리.

### 4.3 API
- Firestore 사용 시: 클라이언트에서 **Firestore SDK**로 직접 읽기/쓰기 가능.  
- 보안이 중요해지면 **Cloud Functions**로 API 레이어를 두고, 인증·권한 검사 후 Firestore/Cloud SQL 접근.

## 5. 관리자 대시보드 요구사항

### 5.1 공통
- **Admin 전용 로그인** (이메일/비밀번호 또는 Google 등).
- 관리자만 접근 가능한 URL(예: `/admin`) 및 역할 체크.

### 5.2 기능 목록

| 영역 | 기능 | 비고 |
|------|------|------|
| **상품 관리** | 상품 CRUD, 카테고리별/브랜드별 필터·정렬 | 상세페이지 편집 포함 |
| **상세페이지** | 상품별 상세페이지 내용·이미지·가격 등 편집 | 아래 “상세페이지 에이전트” 참고 |
| **유저 관리** | 크리에이터/바이어/브랜드 목록·상태·권한 | 검색·필터 |
| **주문 관리** | 주문 목록, 인보이스 발행, 결제·배송 상태 변경 | PO·협상 상태 포함 |
| **게시판** | 공지·FAQ 등 게시판 CRUD | |
| **CRM** | 문의·협상·이벤트 기록 (연락처·메모) | 단계적으로 확장 |
| **통계** | 브랜드 수, 상품 수, 방문자 수, 주문 수, 매출 수량·금액 | 대시보드 위젯·간단 차트 |
| **법률·약관** | 이용약관, 개인정보처리방침 등 등록·노출 관리 | 버전·공개일 관리 |

### 5.3 상세페이지 작업용 “에이전트” 제안
- **상세페이지**가 상품 수백 개 단위로 많고, 문구·이미지 정리가 반복적이면 **전용 에이전트(봇)**를 두는 것이 효율적.
- **역할**:  
  - 입력: 상품명·카테고리·스펙 요약(또는 기존 시트 데이터).  
  - 출력: 상세페이지용 문구(한글), 필요 시 이미지 가이드(어떤 구도/텍스트 넣을지).  
- **도구 예**:  
  - **Cursor + 프로젝트 내 “상품 상세 생성” 스크립트**: 프롬프트에 상품 정보 넣고, 생성된 문구를 DB/대시보드에 붙여넣기.  
  - **NotebookLM**: 상품 목록·브랜드 가이드라인을 넣고 “상세페이지 초안 생성”용으로 사용.  
- **플로우**:  
  1) 관리자가 대시보드에서 상품 선택 → “상세문구 자동 생성” 버튼 클릭.  
  2) 백엔드에서 해당 상품 데이터를 AI(또는 Cursor/NotebookLM 연동 API)에 넘김.  
  3) 생성된 문구를 편집 가능한 폼으로 보여주고 저장.  
  → 1인 개발이면 1단계는 “수동 복붙”으로 시작하고, 2단계부터 API 연동을 붙여도 됨.

## 6. 프론트엔드 요구사항

### 6.1 페이지 구성 (개념)
- **메인(랜딩)**: 서비스 소개, 사용자 유형(크리에이터/바이어/브랜드)별 CTA, 최신 상품 또는 카테고리 링크.
- **서브**:  
  - 상품 목록·검색·필터 (카테고리/브랜드).  
  - 상품 상세.  
  - 크리에이터: 판매 의사 신청·협상·주문.  
  - 바이어: PO 작성·주문.  
  - 브랜드: 검수 요청·신상 요청 (차후: 크리에이터 검색·제안).  
  - 공지·FAQ·약관.

### 6.2 메뉴·디자인·컨텐츠
- **메뉴**: 사용자 유형별로 “크리에이터용 / 바이어용 / 브랜드용” 구분 또는 탭으로 진입점 제공.
- **디자인**: 반응형(모바일 우선), 브랜드 컬러·폰트 일관성, CTA 버튼 명확히.
- **컨텐츠**: 한글 우선, 핵심 문구는 나중에 다국어 키로 치환 가능하게 구조화.
- **사용자별 노출**:  
  - 비로그인: 메인·상품 목록·상세 일부.  
  - 로그인(크리에이터/바이어/브랜드): 각자 플로우(협상, PO, 검수 요청 등) 메뉴 노출.

## 7. AI·개발 도구 추천 (초보자용)

### 7.1 도구별 역할·장단점

| 도구 | 추천 용도 | 장점 | 단점 |
|------|-----------|------|------|
| **Cursor** | 실제 코드 작성(프론트/백/스크립트), 리팩터링, DB 스키마 설계 | 코드베이스 이해·수정에 강함, 터미널·파일 직접 조작 | 구독 비용, 학습 곡선 |
| **ChatGPT** | 기획 문구, API 설계 질문, 샘플 코드·SQL 문 생성 | 범용, 설명이 자세함 | 프로젝트 파일 직접 못 봄 |
| **Gemini** | 문서 요약, 다국어 초안, 스프레드시트·문서 기반 기획 | Google 연동, 긴 문서 처리 | 코드 실행·프로젝트 연동 제한 |
| **Google AI Studio** | API 연동용(챗봇, 상세문구 생성 등) 프로토타입 | API 키로 서비스에 붙이기 쉬움 | 코딩 필요 |
| **NotebookLM** | PRD·상품 목록 업로드 후 “상세문구/기획 Q&A” | 문서 기반 추론, 요약·초안 | 코드·배포와는 분리 |
| **Google Antigravity** | (제품 성격에 따라) 앱/웹 프로토타입 | Google 생태계 | 정보 제한적, 용도 확인 필요 |
| **OPAL** | (용도 확인 후) 작업 자동화·연동 | — | 정확한 역할은 공식 문서 참고 권장 |
| **MCP(Model Context Protocol)** | Cursor 등에 DB·API·외부 도구 연결 | 데이터·도구를 AI 맥락에 포함 | 설정 필요 |

### 7.2 프로젝트별 추천 조합
- **기획·PRD·문서**: Gemini + NotebookLM (문서 정리, 다국어 초안).  
- **실제 개발**: **Cursor** 중심 (코드 생성·수정·디버깅, DB 설계 지원).  
- **상세문구·CRM 문구**: ChatGPT 또는 Google AI Studio API (나중에 “상세 생성” 버튼 연동).  
- **DB·시트 연동**: Cursor로 스크립트 작성 + Google Cloud 콘솔에서 Firestore/Cloud SQL 설정.

### 7.3 MCP 활용
- Firestore/Cloud SQL 접근 MCP가 있으면 Cursor에서 “스키마 보기·샘플 쿼리” 요청 가능.  
- Slack·이메일 알림 MCP로 “주문 접수·협상 요청” 알림 자동화 가능 (차후).

## 9. 개선 제안 요약

- **순서**: 기획 → 백엔드(Product DB) → 관리자 대시보드 → 퍼블릭 프론트엔드 → 다국어·앱.  
- **DB**: 1인·초기에는 Firestore로 상품·계정·주문 모두 시작해도 되고, 주문/결제가 복잡해지면 Cloud SQL 도입 검토.  
- **상세페이지**: 상품 수 많으면 “상세문구 생성” 에이전트(또는 Cursor/NotebookLM 활용)로 초안 생성 후 대시보드에서 편집·저장.  
- **AI**: Cursor(개발) + Gemini/NotebookLM(기획·문서) + ChatGPT/Google AI Studio(문구·API 설계) 조합 권장.

## 10. 초보자용 실행 가이드 (바이브코딩 팁)

### 10.1 한 번에 다 하지 않기
- **MVP(최소 기능)** 먼저: “상품 목록만 보이는 랜딩 + 관리자 상품 등록”만 만들어도 1차 목표 달성.
- 크리에이터 협상·바이어 PO·브랜드 검수는 **2차 단계**로 나누어 추가.

### 10.2 Cursor에서 할 때
- 할 일을 **한 문장으로** 적고 요청 예: “Firestore에 상품 컬렉션 만드는 코드 추가해줘”.
- 에러가 나면 **에러 메시지 전체**를 복사해 붙여넣고 “이걸 해결해줘”라고 요청.
- 파일 이름·폴더 구조를 미리 정해 두면 AI가 그에 맞춰 코드를 제안하기 쉬움.

### 10.3 DB 데이터 넣기 (수백 개 상품)
1. 엑셀/스프레드시트에 상품 목록 정리 (이름, 카테고리, 브랜드, 가격 등).
2. CSV 또는 JSON으로 저장.
3. Cursor에게 “이 CSV를 Firestore에 넣는 스크립트 만들어줘”라고 요청 후 실행.

### 10.4 상세페이지 에이전트 만드는 법 (요약)
- **옵션 A**: Cursor 프로젝트 안에 `scripts/generate-product-description.js` 같은 파일을 만들고, 상품 정보를 인자로 받아 ChatGPT/Gemini API를 호출해 문구 생성 → 결과를 복사해 대시보드에 붙여넣기.
- **옵션 B**: NotebookLM에 “상품 가이드·톤앤매너” 문서를 넣고, 상품명만 주고 “상세페이지 초안 달라”라고 질의한 뒤 결과를 수동 반영.
- 나중에 “상세 생성” 버튼을 대시보드에 붙일 때는 옵션 A를 백엔드(Cloud Functions 등)에 연결하면 됨.

## 12. 용어 정리 (초보자용)

| 용어 | 설명 |
|------|------|
| **PRD** | 제품 요구사항 정의서. “무엇을, 왜, 누가 쓰는지”를 적은 문서. |
| **DB** | 데이터베이스. 상품·회원·주문 정보를 체계적으로 저장하는 곳. |
| **API** | 프로그램끼리 데이터를 주고받는 규칙·경로. |
| **CRUD** | 만들기(Create), 읽기(Read), 수정(Update), 삭제(Delete). |
| **PO** | Purchase Order. 구매 주문서. |
| **MVP** | Minimum Viable Product. 동작하는 최소 기능만 갖춘 버전. |
| **i18n** | 다국어 지원 (internationalization). |
| **MCP** | AI가 외부 도구·DB와 연결할 때 쓰는 프로토콜. |
